# App Flow Document for TradingView-API

## Onboarding and Sign-In/Sign-Up
A developer who wants to use the TradingView-API first clones the repository or installs it from npm. They create a TradingView account on the official TradingView website and obtain their API credentials. Back in their local environment, they copy the provided `.env.example` file into a new `.env` file and fill in the required variables, such as username, password or API key, and any optional settings like default symbol or timeframe. Once the environment variables are set, they install the dependencies by running `npm install` in the project folder. To sign in programmatically, the developer calls the client’s `login` method, passing their credentials; on successful authentication, the WebSocket connection is established and the session token is managed automatically. If the login fails due to an incorrect password or expired account, the developer is prompted to recover their password through the TradingView web portal and then retry the login. Signing out happens by calling a `logout` function or by closing the Node.js process, which in turn closes the WebSocket and clears the session from memory.

## Main Dashboard or Home Page
After successful login, the developer sees the default client instance that has connected to TradingView. In code, this is represented by a `Client` object that emits an `onReady` event to signal that the connection is open and authenticated. Although there is no graphical dashboard, the `Client` object serves as the central hub. It exposes namespaces for quotes, charts, and indicators, each accessible by calling methods such as `quoteSession` or `chartSession`. From this initial ready state, the developer can navigate programmatically to different parts of the API by invoking factory methods on the client. For example, they call `client.createQuoteSession()` to start a real-time data feed or `client.createChartSession()` to open a new chart context. The developer can choose to listen for events or call direct methods, making the API feel like a dashboard of capabilities in code form.

## Detailed Feature Flows and Page Transitions

When the developer wants real-time market data, they request a quote session through the client. The quote session connects over WebSocket, subscribes to a symbol, and then emits `onUpdate` events containing the latest price and volume information. The developer writes an asynchronous callback to handle each update, updating their own data structures or triggering downstream calculations.

To integrate technical indicators, the developer first creates an indicator object—either a built-in indicator or a custom Pine Script strategy—by invoking the client’s indicator factory method. They pass any required inputs such as length or source series. Once configured, the indicator object subscribes to the chart session or quote session and emits its own `onData` updates. This lets the developer chain multiple indicators or combine them with real-time quotes.

Chart sessions are managed similarly. The developer calls `createChartSession` and provides parameters such as symbol, timeframe, and chart type. The chart session establishes its own event listeners for symbol load and timeframe change. The developer can then attach studies or annotations by passing the chart session reference into indicator objects. The code flow moves seamlessly from chart creation to study attachment, with each step acknowledged by an event or callback.

For advanced Pine Script workflows, the developer uses a Pine indicator class that handles uploading or updating script code. They call a method to set script inputs and request permission scopes from the API. Once the script is live, the client listens for `onScriptUpdate` events so the developer can inspect strategy performance or backtesting results. This entire process is handled via method calls and event hooks, ensuring the journey from script deployment to live data analysis is fully automated.

Automated trading and alerts rely on subscribing to specific market conditions. The developer writes logic that listens to price and indicator events. When a certain threshold is crossed, their callback triggers either an alert or a trade execution call. This flow moves from data subscription to decision logic and then to trade or notification requests, all orchestrated within a single Node.js process without manual intervention.

## Settings and Account Management
The developer can adjust preferences by editing the `.env` file or by passing configuration options directly into the client constructor. This includes setting default symbols, timeframes, connection retry limits, or logging verbosity. For subscription billing or quota details, the developer refers to their TradingView account dashboard on the TradingView website. Within the code, they can query the client for current usage or subscription level, which the session manager exposes through a `getSubscriptionInfo` method. After changing any settings, the developer restarts the Node.js process to reload the configuration. They then return to the main workflow by logging in again and calling the desired session creation methods.

## Error States and Alternate Paths
If the developer enters invalid credentials, the `login` call rejects and emits an `onError` event with a clear message. The client then disconnects, and the developer must correct their `.env` file or TradingView password. In the case of lost network connectivity, the WebSocket automatically attempts reconnection a configurable number of times. During this period, the client emits `onReconnectAttempt` and `onReconnectError` events. If the maximum retry count is reached, the client gives up and emits a final `onDisconnect` event, at which point the developer can choose to restart the process or handle fallback data. If the developer tries to subscribe to a symbol that does not exist or that they do not have permission for, the session returns an immediate error callback explaining the problem and cleans up the failed subscription. All error messages provide enough context so the developer can correct the issue and resume the normal flow.

## Conclusion and Overall App Journey
A typical journey with the TradingView-API starts when a developer signs up for TradingView, sets up credentials locally, and installs the library. They log in programmatically and see their client reach a ready state. From there, they choose to fetch live quotes, integrate one or more technical indicators, and manage chart sessions without manual chart interactions. Advanced users deploy Pine scripts and automate entire trading strategies or alert systems. When preferences change, they update configuration and restart the application. Along the way, robust error handling ensures that invalid credentials, network issues, or permission errors are caught and reported. This end-to-end flow empowers developers to automate market analysis and trading tasks entirely within a Node.js environment.
